# 01 — Security Rules

> Constraint context: fully static site, hosted free on Cloudflare Pages, no server, no paid services.

## 1. Secrets & keys

- **No secret ever enters the repo or the browser.** API keys, tokens, and credentials live **only** in a developer's local `.env` file.
- `.env` (and `.env.*` except `.env.example`) **must be gitignored**. Verify before every commit.
- Provide a committed `.env.example` listing required variable **names only** (no values) so devs know what to set.
- Secrets are used **only** at build/sync time on a developer's machine — never read at runtime by the live site.

## 2. What is public vs. protected (static-site reality)

- **Public (unavoidable):** anything rendered on a page — WhatsApp number, review text, follower counts, image/video URLs. Do not pretend these are hidden.
- **Protected:** API keys/tokens (kept in local `.env`), and any data we simply choose not to publish.
- Rule: never put anything in client code that would be damaging if seen by a visitor. If it must stay private, it does not belong on a static site.

## 3. Central config

- WhatsApp number, brand info, and social handles live in **one** committed file: `content/site.config.yaml`.
- Social snapshot data lives in **one** committed file: `content/data/social.json` (generated by the sync script — see [04-social-sync.md](04-social-sync.md)).
- Do not scatter these values across components.

## 4. Outbound links & user input

- WhatsApp links use `https://wa.me/<number>?text=<encoded>`; always `encodeURIComponent` the prefilled text.
- All external links use `rel="noopener noreferrer"` and `target="_blank"` only when needed.
- Enquiry forms are **client-only** (build a WhatsApp message); no data is posted to any server. Sanitize/encode user-entered values before placing them in a URL.

## 5. HTTP security headers (Cloudflare `public/_headers`)

Keep the hardened defaults already in `public/_headers`:

- `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`.
- `Strict-Transport-Security` enabled.
- **Content-Security-Policy** scoped to only the origins we actually use. When adding a new external resource (font, embed, script), update the CSP allow-list deliberately — do not use wildcards.

## 6. Dependencies

- Prefer zero/low dependencies. Every new package must be free & open-source (MIT/Apache/BSD).
- No package that requires a paid plan, account, or phone-home telemetry.
- Run `npm audit` before release; fix high/critical issues.
- Lockfile (`package-lock.json`) is committed.

## 7. Pre-commit checklist

- [ ] No `.env` / secret staged.
- [ ] No hardcoded key, token, or password anywhere.
- [ ] New external origins added to CSP if required.
- [ ] No data posted to a third-party server.
